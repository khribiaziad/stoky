Products

Catalog management — products, variants, and packs
Products are the top-level items sold. Each product has one or more Variants which represent specific size/color combinations. Stock is tracked at the variant level. Packs are bundles of multiple variants sold as a single unit at a fixed price.
Create Product POST /api/products POST
Inputs
  • required name — product name (string)
  • optional category — default "caps"
  • optional has_sizes — bool (default true)
  • optional has_colors — bool (default true)
  • optional under_1kg — bool (default false). If true, "Sell Bag" is auto-checked on any order containing this product
Process
  1. Validates name is not empty
  2. Creates Product record linked to current user's store
  3. Returns the new product (no variants yet)
DB Impact
INSERT products
After creating a product, user must immediately add at least one variant before the product can be used in any order. There is no "draft" state — an empty product with no variants is a dead end in the UI.
Add Variant POST /api/products/{id}/variants POST
Inputs
  • optional size — e.g. "M", "L", "XL"
  • optional color — e.g. "Black", "Red"
  • required buying_price — cost per unit (MAD)
  • optional selling_price — default selling price (used for reference; actual price comes from the order's total_amount)
  • optional stock — initial stock qty (default 0)
  • optional low_stock_threshold — alert threshold (default 5). Alert shows on Dashboard when stock ≤ this value.
Process
  1. Validates product exists and belongs to current store
  2. Creates Variant record linked to product
  3. Returns updated product with all variants
DB Impact
INSERT variants
Setting initial stock here is confusing — ideally stock always comes from a StockArrival record (for auditability). Direct stock entry on variant creation bypasses the stock history log.
Edit Variant PUT /api/products/variants/{id} PUT
Inputs
  • Same fields as Add Variant
  • ⚠ affects stock If stock value is changed here, it's a direct override with no audit trail
Process
  1. Finds variant by ID
  2. Updates all provided fields in-place
  3. Returns updated product
DB Impact
UPDATE variants
Changing buying_price on a variant does NOT retroactively update past order items. OrderItem records store a snapshot of unit_cost at creation time — so historical profit calculations remain correct, but the user gets no warning that old orders used a different cost.
Delete Variant / Delete Product DELETE /api/products/variants/{id}  |  DELETE /api/products/{id} DELETE
What happens
  1. Deleting a variant: removes the variant. If linked to pending orders, those OrderItems lose their variant reference (variant_id becomes null). Past order data (product_name, unit_cost) is preserved as a snapshot.
  2. Deleting a product: cascades to delete all variants and their relationships.
DB Impact
DELETE variants DELETE products
No soft-delete / archive

Products are permanently deleted. There is no way to deactivate a product without deleting it.

No confirmation of how many orders reference the variant before deleting. No archive/hide option — if a product is discontinued, the only option is to delete it, which removes it from the product list in historical order views.
Create Pack POST /api/packs POST
Inputs
  • required name — pack name
  • required selling_price — total selling price of the pack
  • required item_count — how many items are in the pack (drives how many product selectors appear)
What it is

A pack is a bundle sold at a single price. When assigned to an order, the user selects specific variants to fill each slot in the pack. The pack's component variants are each logged as separate OrderItems, so stock is deducted per-variant. Presets are named preset compositions (e.g. "S/M/L assortment") that auto-fill the slots.

Packs do NOT have their own stock. Stock is consumed from the component variants when orders are saved.
Add Pack Preset POST /api/packs/{id}/presets POST
Inputs
  • required name — preset name (e.g. "S+M+L mix")
  • required items — array of {variant_id, quantity}
Purpose

Presets save frequently-used pack compositions so the user can one-click fill all variant slots instead of selecting each one manually. When selected, auto-fills the pack's product slots in the order assignment screen.

Presets are pack-level only. If a user always ships the same single-product variant in most orders, there is no "default variant" shortcut for regular (non-pack) products.

Orders

The core daily workflow — creating, managing, and exporting orders
Orders are created one of two ways: (1) uploading a Pickup PDF from the carrier (most common), or (2) manually entering an order. Every order has a CMD-ID (carrier tracking number), customer info, one or more product items, per-order expenses, and a status (pending / delivered / cancelled).
Upload Pickup PDF POST /api/orders/upload-pickup POST
Input
  • required A PDF file from the carrier (Caleo, Amana, etc.)
  • The PDF contains: CMD-ID, customer name, phone, address, city, total amount (COD)
  • The PDF does NOT contain which product was ordered
What gets parsed
  • pdfplumber extracts tables or raw text from each page
  • Regex patterns match CMD-IDs (e.g. "CMD-123456")
  • Customer name, phone, city, address, amount extracted per order row
Process steps
  1. PDF saved to /uploads/ directory
  2. parse_pickup_pdf() runs synchronously (blocking)
  3. Returns array of extracted order data (not yet saved to DB)
  4. Frontend shows the "Assign Products" modal
Output

A preview list of orders extracted from the PDF. Nothing is saved to the database yet — this is just a parse result displayed to the user for review.

This step is synchronous and can time out on large PDFs. The parser is also format-sensitive — if the carrier changes their PDF layout, rows may be skipped or misread with no warning. There is no validation step showing the user exactly what was/wasn't parsed before they proceed.
Assign Products to Orders UI ONLY
What happens

After the Pickup PDF is parsed, the user sees a modal with one card per order. For each order, they must:

  1. Choose "Product" or "Pack" mode for each line item
  2. Select a variant from a dropdown (shows current stock)
  3. Set quantity
  4. Optionally add more items to the same order (+ Add Item)
  5. Set per-order expenses: Sticker (1 MAD), Sell Bag (1 MAD, auto-checked if product is under 1kg), Packaging (MAD, editable)
Friction points
  • If a PDF has 40 orders and all contain the same product, the user must select that product from the dropdown 40 times
  • No "apply to all" or "same as above" shortcut
  • No stock quantity validation warning until Save is clicked
  • The dropdown shows ALL variants across all products — can be very long with no search/filter
  • Sell Bag auto-checks only works correctly if the product is flagged — often missed at product creation
  • There is no way to skip an order or flag it for later
This is the highest-friction step in the entire app. For a batch of 50 orders, assigning products one by one in a modal with large dropdowns is the main daily bottleneck. The entire modal must stay open until all orders are assigned — there is no partial save.
Save Orders (Bulk Create) POST /api/orders/bulk-create POST
Input
  • Array of order objects, each containing: caleo_id, customer info, total_amount, items[], expenses{}
  • Orders with no items assigned are automatically skipped
Process (per order)
  1. Checks if CMD-ID already exists for this store (prevents duplicate imports)
  2. Looks up city in City table → gets delivery_fee and return_fee
  3. If city not found, uses fallback fees from AppSettings
  4. Creates Order record (status = pending)
  5. For each item: validates variant exists, checks stock is sufficient, deducts stock, creates OrderItem
  6. Creates OrderExpense record with sticker, seal_bag, packaging, delivery_fee, return_fee
DB Impact
INSERT orders INSERT order_items INSERT order_expenses UPDATE variants.stock (−qty) SELECT cities
On failure

If any variant has insufficient stock, the entire batch rolls back. The user sees an error naming the specific variant and available stock. They must fix the assignment and retry the entire batch.

City name matching is case-insensitive but exact otherwise. "Casablanca" ≠ "Casa" ≠ "CASA" — if the carrier writes the city differently in their PDF, the order gets the fallback fee instead of the correct one, silently affecting profit calculations.
Create Manual Order POST /api/orders/bulk-create (single item) POST
Inputs
  • required CMD ID, Customer Name, Total Amount
  • optional Phone, City, Address
  • required At least one product (variant + qty)
  • Expenses: sticker, seal_bag, packaging
Use case

Used when an order was created outside the normal carrier flow (e.g. Instagram DM, phone sale, or a carrier order that didn't appear in the PDF). Follows the same backend logic as bulk-create but with a single order.

The user must manually type the CMD-ID, which must be unique. If they enter a CMD-ID that already exists in the system, the order is silently skipped (not saved, no error shown — it's treated as "already imported"). This can confuse users who don't realise the silent skip.
Edit Order PUT /api/orders/{id} PUT
Editable fields
  • CMD ID, Customer Name, Phone, City, Address, Total Amount
  • Items (full replacement — old items deleted, new ones added)
  • Expenses (sticker, seal_bag, packaging)
Stock rules
  • Pending orders: old items' stock is restored, new items' stock is deducted
  • Delivered orders: items updated in DB, stock NOT touched (already consumed)
  • Cancelled orders: items updated in DB, stock NOT touched (already restored)
City change

If city is changed, delivery_fee and return_fee on the OrderExpense record are automatically recalculated from the City table.

DB Impact
UPDATE orders DELETE order_items (old) INSERT order_items (new) UPDATE variants.stock UPDATE order_expenses
Change Order Status (individual) PUT /api/orders/{id}/status?status=... PUT
States
  • pending → waiting for delivery outcome
  • delivered → revenue counted, stock gone
  • cancelled → return fee counted, no revenue
Important

Status change via this dropdown does NOT restore stock when setting to cancelled, and does NOT trigger the broken stock or seal_bag return logic. Only the Return flow (PDF or manual) does that. This dropdown is a simple metadata update.

The distinction between "changing status to cancelled via dropdown" vs "processing a return via the return flow" is unclear to users. The dropdown change is a simple label change; the return flow actually affects stock and profit. Users who cancel orders via the dropdown bypass the full return accounting.
Bulk Status Update POST /api/orders/bulk-status POST
Inputs
  • required order_ids — array of order IDs (from checkboxes)
  • required status — "pending", "delivered", or "cancelled"
Process
  1. Validates status value
  2. Updates all matched orders' status in one DB call
  3. Returns count of updated orders

Same caveat as individual status change: this is a label update only, no stock or return-fee logic runs.

Bulk marking as "delivered" is useful for confirming many orders at once (e.g. when the carrier sends a delivery confirmation report). However, there is no bulk return processing — bulk-cancelled orders bypass the return accounting (stock restoration, broken stock logging).
Add / Edit Order Notes PATCH /api/orders/{id}/notes PATCH
Input
  • Free-text string stored in orders.notes column
  • Replaces previous notes entirely (no append history)
Use cases
  • Follow-up reminders ("customer said to call again Monday")
  • Special delivery instructions
  • Internal flags ("duplicate CMD, check with carrier")
Notes are visible only when the ✏ button is clicked — they are not shown inline in the table. A user cannot see at a glance that a note exists without clicking each order. The amber 📝 icon helps, but notes are not searchable.
Export Orders to CSV UI ONLY (client-side)
What it exports

Exports the currently visible (filtered) orders. Columns: CMD-ID, Customer, Phone, City, Address, Amount, Status, Items, Confirmed By, Date, Notes.

The CSV is generated entirely in the browser from the already-loaded orders state — no extra API call. Uses BOM (UTF-8 with \ufeff) for correct display in Excel.

Respects filters
  • Status filter (all / pending / delivered / cancelled)
  • Search text (name, CMD, city, phone)
  • Combined filters
Delete Order DELETE /api/orders/{id} DELETE
Process
  1. If order is pending: restores all item quantities back to variant stock
  2. If delivered or cancelled: no stock change
  3. Cascades to delete OrderItems and OrderExpenses
DB Impact
DELETE orders DELETE order_items DELETE order_expenses UPDATE variants.stock (+qty, if pending)
No soft-delete. Deleted orders disappear from all reports. If an order was delivered and then deleted, that revenue is gone from history with no audit trail.
Search & Filter Orders UI ONLY (client-side)
Filter by status

Tab buttons: All / Pending / Delivered / Cancelled. Filters in-memory from loaded orders.

Search matches
  • Customer name (case-insensitive substring)
  • CMD-ID (substring)
  • City (case-insensitive substring)
  • Phone number (substring — useful for customer lookup)
Limitation

All orders for the store are loaded into memory at once on page load. With thousands of orders this becomes slow and the initial load gets heavy. There is no server-side pagination or lazy loading.

No date range filter. No filter by product, city group, or confirmer. No sort controls (orders always show newest first by order_date). As the orders table grows, this page will become harder to use.

Returns

Processing undelivered parcels returned by the carrier
Upload Return PDF POST /api/orders/upload-return POST
Input

A PDF from the carrier listing CMD-IDs of returned (undelivered) parcels.

Process
  1. PDF saved to /uploads/
  2. parse_return_pdf() extracts CMD-IDs from the document
  3. Each CMD-ID is looked up in the orders table
  4. Returns: matched_orders (found in DB) + unmatched_cmd_ids (not found)
Output modal

User sees a list of matched orders. For each, they set two flags:

  • Sell Bag Returned — carrier returned the packaging bag (adds back 1 MAD cost recovery)
  • Product Broken — product damaged in transit (stock goes to broken log, not restored)

Unmatched CMD-IDs are shown as an error — usually means the order wasn't imported yet or was already cancelled.

Unmatched CMD-IDs are displayed but there is no action the user can take from that screen — they must close the modal, find the order (or create it), and then process the return manually. The unmatched list is not saved anywhere.
Process Returns (confirm) POST /api/orders/process-returns POST
Input
  • Array of: order_id, seal_bag_returned (bool), product_broken (bool)
Process (per return)
  1. Sets order.status = "cancelled"
  2. Saves seal_bag_returned and product_broken flags on OrderExpense
  3. If product_broken = true: creates BrokenStock record; does NOT restore variant stock
  4. If product_broken = false: restores variant.stock += item.quantity for each item
DB Impact
UPDATE orders.status UPDATE order_expenses UPDATE variants.stock (+qty, if not broken) INSERT broken_stock (if broken)
Financial impact

Cancelled orders contribute return_fee (cost) and no revenue to reports. If seal_bag_returned = true, the cost is slightly offset. Broken stock value_lost is logged at buying_price × quantity.

Manual Return POST /api/orders/process-returns (single item) POST
Use case

For processing a single return not captured in a PDF (e.g. customer brings back product directly, or a carrier return that wasn't in the batch PDF).

UI flow
  1. User searches for the order by CMD-ID or customer name
  2. Only pending orders are shown (can't return an already-cancelled order)
  3. User selects the order, sets the two flags, confirms
  4. Same backend logic as PDF-batch returns
Searching only shows pending orders. If an order was mistakenly marked delivered via status dropdown and then needs to be returned, the user cannot find it in the manual return search — they must first change it back to pending, then process the return.

Stock

Inventory arrivals and broken stock logging
Add Stock Arrival POST /api/stock/arrivals (bulk array) POST
Inputs (per line)
  • required variant_id
  • required quantity
  • required total_cost — total paid for this batch (MAD)
  • optional additional_fees — import/shipping fees on top
  • optional description — supplier notes
  • required date
Process
  1. Creates StockArrival record for audit history
  2. Adds quantity to variants.stock
  3. Creates a Withdrawal record of type "stock_purchase" for the total cost paid (so stock purchases appear in the capital/cash flow)
DB Impact
INSERT stock_arrivals UPDATE variants.stock (+qty) INSERT withdrawals (type=stock_purchase)
Buying price on the Variant record is set once at creation and not updated when new stock arrives at a different cost. If the supplier raises prices, the user must manually edit the variant's buying_price separately — the arrival cost doesn't feed back into the variant's unit cost used in profit calculations.
Log Broken Stock (manual) POST /api/stock/broken POST
Inputs
  • required variant_id, quantity, date
  • optional returnable_to_supplier (bool)
  • source is always "storage" for manual entries (vs "return" for return-flow)
  • value_lost is calculated as variant.buying_price × quantity
Process
  1. Creates BrokenStock record
  2. Does NOT automatically deduct from variant.stock — this is a known gap. The stock count on the variant does not reflect broken units unless the user also manually adjusts the variant's stock number.
Broken stock from returns does deduct from stock (it was never restored). But manual broken stock logging does NOT deduct from stock. This means variant.stock can show 10 units even if 2 are logged as broken in storage — the broken stock log and the variant stock counter can be out of sync.

Expenses

Fixed costs, ad spend, team costs, and cash withdrawals
Create / Edit Fixed Expense POST /api/expenses/fixed  |  PUT /api/expenses/fixed/{id} POST/PUT
Inputs
  • required name, type, amount
  • type options: monthly, annual, per_order, one_time
  • category: operations, packaging, platform, software, equipment, legal, marketing, other
  • optional description, start_date
How each type affects reports
  • monthly — full amount deducted each month in report period
  • annual — amount ÷ 12 deducted per month
  • per_order — amount × delivered_orders_count deducted
  • one_time — counted in all-time totals, not prorated by period
Monthly and annual expenses are not prorated within a partial month. If a report covers "this week", the full monthly amount is still deducted — this over-represents costs for short periods and makes weekly profit look worse than it is.
Toggle Fixed Expense Active/Paused PATCH /api/expenses/fixed/{id}/toggle PATCH
What it does

Flips is_active boolean. Inactive expenses are excluded from monthly burn rate and profit calculations. Useful for seasonal expenses or temporarily suspended services.

UI display

Inactive expenses are shown at 45% opacity in the Fixed Expenses tab. They remain visible and can be reactivated.

Ad Platform + Ad Campaign POST /api/expenses/platforms  |  POST /api/expenses/campaigns POST
Platform inputs
  • name (slug), label (display name), color
  • Examples: Facebook/Meta, TikTok Ads, Google Ads
Campaign inputs
  • daily_rate_usd — daily budget in USD
  • start_date — when campaign started
  • end_date — leave blank for currently running campaign
How costs are calculated
  1. For each campaign, count the number of days that overlap with the report period
  2. Multiply: days × daily_rate_usd × usd_exchange_rate
  3. Sum across all campaigns on all platforms
USD rate

Stored in AppSettings as "usd_rate" (default 10 MAD). Set in Settings page. Applied globally to all campaign cost calculations.

Daily rate is a flat estimate — it doesn't reflect actual spend which varies day to day. There is no way to import actual spend from Facebook/TikTok dashboards. If a campaign's actual daily spend was different from the set rate, reports are inaccurate. No monthly budget caps or alerts.
Add Withdrawal POST /api/expenses/withdrawals POST
Types
  • manual — owner takes cash out. Created here by the user.
  • stock_purchase — automatically created when a Stock Arrival is logged. Cannot be created manually.
Capital impact

Withdrawals reduce the Cash Balance in the capital calculation:
cash = revenue_collected − expenses − withdrawals
They represent money that has physically left the business.

Team

Staff management, confirmer accounts, and performance stats
Create Team Member POST /api/team POST
Inputs
  • required name
  • optional role — job title
  • required payment_type — "monthly", "per_order", or "both"
  • optional fixed_monthly — MAD/month salary
  • optional per_order_rate — MAD/delivered order commission
  • optional start_date, end_date
How it affects reports
  • monthly salary is added to monthly burn rate regardless of period
  • per_order_rate × delivered orders in period is deducted from profit
  • A member with payment_type "both" contributes both costs
Create Confirmer Account POST /api/team/{member_id}/create-account POST
Inputs
  • username, password
  • The team member must exist first
What gets created
  1. Creates a User record with role="confirmer"
  2. Links it to the admin's store via store_id
  3. Links it to the TeamMember record via team_member_id
  4. Orders uploaded by this confirmer are tagged with their user_id
What confirmers can do
  • Upload Pickup PDFs → create orders (tagged as their own)
  • Process returns on their own orders
  • View only their own uploaded orders
  • See their own stats (orders confirmed, earnings, delivery rate)
What they cannot do
  • View full order list, finances, reports, expenses, settings
  • Edit products, stock, packs, cities
  • See other confirmers' orders
A confirmer can only be linked to one team member record. If the same person works multiple roles, they need separate accounts. There is no way to give a confirmer read-only access to the full order list (e.g. a manager who needs to monitor all orders but not edit anything).
View Member Stats GET /api/team/{member_id}/stats?period=... GET
Returns
  • Total orders confirmed by this member
  • Delivered / Pending / Cancelled breakdown
  • Delivery rate %
  • Earnings = delivered_orders × per_order_rate
Period options

Same period options as the Dashboard: today, yesterday, this_week, last_7_days, this_month, all.

Suspend / Reactivate Confirmer Account POST /api/team/{member_id}/toggle-account POST
What it does

Toggles users.is_approved on the linked User record. When false, the confirmer's login is rejected at the JWT auth step. All their data (orders, history) is preserved and remains visible to the admin.

Use case

Used when a team member leaves or is temporarily suspended. A clean way to revoke access without deleting the account or losing historical data.

Reports

Financial summary, performance metrics, period comparison
Load Report Summary GET /api/reports/summary?period=...&start=...&end=... GET
Period options
  • today, yesterday, this_week, last_7_days, this_month
  • empty = all time
  • start + end = custom date range (ISO dates)
Returns

financials: revenue, gross_profit, clean_profit, team_costs, fixed_costs, ads_costs

capital: total_capital, cash_balance, stock_value

orders: total, delivered, pending, cancelled, delivery_rate, return_rate, avg_order_value

How profit is calculated
  1. Revenue = sum of total_amount on delivered orders in period
  2. COGS = sum of (unit_cost × quantity) on items of delivered orders
  3. Per-order costs = delivery_fee + packaging + sticker + seal_bag (minus seal_bag_returned) on delivered orders
  4. Return costs = return_fee on cancelled orders in period
  5. Gross profit = Revenue − COGS − per-order costs − return costs
  6. Team costs = monthly salaries + (per_order_rate × delivered orders)
  7. Fixed costs = active monthly expenses + annual÷12 + per_order×delivered + one_time
  8. Ad costs = days_in_period × daily_rate_usd × usd_rate (per active campaign)
  9. Clean profit = Gross − team − fixed − ads
Capital (total_capital, cash_balance, stock_value) is always current — it ignores the selected period. Choosing "This Month" shows this month's revenue but the current real-time capital, not the capital at the end of that period. This makes historical period comparisons of capital meaningless.
Period Comparison UI ONLY (second API call)
How it works
  1. User clicks "⇄ Compare" toggle
  2. Frontend computes previous period start/end dates (e.g. "this month" → last month's dates)
  3. Makes a second call to GET /api/reports/summary with those custom dates
  4. Shows ▲/▼ delta next to Revenue, Gross Profit, Clean Profit stat cards
Previous period mapping
  • today → yesterday
  • yesterday → 2 days ago
  • last_7_days → 7–14 days ago
  • this_week → previous calendar week
  • this_month → previous calendar month
  • all / custom → comparison not available
Print Report UI ONLY
What it does

Calls window.print(). The browser's native print dialog opens. The user can save as PDF or send to a printer. The full page including sidebar and navigation will print unless the browser or user adjusts the settings.

Limitation

No print-specific CSS is applied. The dark sidebar, navigation, and filter controls all print alongside the report data. A proper implementation would need @media print CSS to hide non-report elements.

Settings

Store configuration, delivery fees, city management
Update Store Name PATCH /api/auth/update-store PATCH

Updates the store_name field on the admin User record. The store name is displayed in the sidebar header.

UPDATE users.store_name
Set Delivery Fee Fallback POST /api/settings/default_delivery_fee  &  /api/settings/default_return_fee POST
What it does

Stores two key-value pairs in AppSettings. These are used as fallback delivery_fee and return_fee when an order's city is not found in the City table.

UI behaviour

This section is visually greyed out (opacity 0.45, pointer-events none) when a city list is uploaded, showing an "OVERRIDDEN BY CITY LIST" badge. However, it still activates as a fallback for cities not in the list.

City CRUD (add / edit / delete) POST/PUT/DELETE /api/cities/{id} POST/PUT/DELETE
Fields per city
  • name — must be unique (case-sensitive)
  • delivery_fee — MAD charged by carrier on delivery
  • return_fee — MAD charged by carrier on return
  • is_casa — flag for same-city (Casablanca metro) orders. Currently stored but not used differently in calculations.
How it's used

When an order is created, the system looks up orders.city in the City table using a case-insensitive match. If found, the city's delivery_fee and return_fee are stored on the OrderExpense record. These fees directly affect profit calculation.

Changing a city's fee

Editing a city record does NOT retroactively update existing orders. Past OrderExpense records retain the fee values that were set at order creation time.

Import City Fees via PDF POST /api/cities/upload-pdf → GET /api/cities/pdf-job/{id} POST
Process
  1. PDF uploaded, returns job_id immediately (non-blocking)
  2. Background thread runs pdfplumber on the PDF (up to 150 pages)
  3. Extracts city name + delivery fee + return fee using table detection or text regex
  4. Frontend polls /cities/pdf-job/{id} every 1.5s for progress
  5. Shows "Reading page X of Y…" progress bar
  6. On completion: upserts cities (update if exists, insert if new)
Column detection

Searches table headers for keywords: ville/city/destination/nom → city column; livraison/delivery/tarif → fee column; retour/return → return fee column. Falls back to regex pattern matching on raw text if no table found.

Upsert logic

City names are title-cased. If a city with that name already exists, its fees are updated. If it's new, it's inserted. is_casa defaults to false for all imported cities.

City name normalisation is basic (title-case only). If the PDF has "CASABLANCA" and the DB already has "Casablanca" they will be treated as a match (SQLAlchemy exact string match on the title-cased name), but if there are spelling variants like "Casa Blanca" vs "Casablanca" these will be inserted as two separate cities.

Friction Summary — Highest Priority

# Friction Point Module Impact
1 Manual product assignment per order after PDF upload. No "apply to all", no default, no quick-fill. For 50 orders this is 50 separate dropdown selections minimum. Orders Daily — blocks every pickup batch
2 PDF parsing has no validation preview. If the parser misreads a row (wrong name, wrong amount, missing city), the user only finds out after saving — or never. Orders Daily — data quality risk
3 Status dropdown and Return flow are separate. Marking "cancelled" via dropdown bypasses stock restoration and return-fee accounting. Users may not know which to use. Orders / Returns Daily — profit accuracy risk
4 No date-range filter or pagination on Orders page. All orders load at once. Over time this page will be slow and hard to navigate. Orders Long-term scalability
5 Expenses not prorated for short periods. Full monthly costs shown even for "today" or "this week" reports, making short-period profit look incorrectly negative. Reports Weekly — misleading reports
6 Manual broken stock does not deduct from variant stock. The stock counter and the broken stock log can be out of sync. Stock Inventory accuracy
7 Buying price on variant is static. Changing it doesn't update historical orders. New stock arrivals at a different cost don't update the unit cost used in profit calculations. Products / Stock Profit accuracy over time
8 City name matching is exact (after title-case). Spelling variations in carrier PDFs will silently use the fallback fee instead of the correct city fee. Settings / Orders Daily — silent profit errors
